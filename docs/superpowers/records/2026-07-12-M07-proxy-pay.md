## 2. 设计要点

### 2.1 业务流程

1. 下单员工创建订单时选择 `payment_method=proxy`
2. 员工调用 `POST /orders/{id}/proxy-pay-link` 获取分享链接
3. 将链接分享给同事（App 分享能力留待 M09/M10）
4. 同事打开 H5 落地页，登录后确认代付
5. 后端以 JSAPI 模式创建预支付单，支付回调后更新 `paid_by_user_id`

### 2.2 Token 与有效期

- **Token 存储**：`proxy_pay_tokens` 表，48 字符随机串，每订单可复用未过期 token
- **有效期计算**：`ProxyPayExpiryCalculator` 读取 `order.auto_cancel_minutes`，自订单创建时间起算
- **过期行为**：预览与支付均返回 `42205`（代付链接已过期）

### 2.3 支付人与订单关联

- **Payment 层**：`payments.payer_user_id` 记录实际付款人（代付场景为代付同事）
- **Order 层**：`ConfirmPaymentHandler` 回调时将 `payer_user_id` 写入 `orders.paid_by_user_id`
- **自付场景**：`payer_user_id` 为空时 fallback 为订单所属用户

### 2.4 网关与 JSAPI

- `InitiateProxyPaymentHandler` 固定 `trade_type: JSAPI`，支持传入 `openid`
- `WechatPayGateway` 扩展 JSAPI 预下单参数
- 开发环境默认 `channel=fake`；H5 页模拟 notify 完成闭环

### 2.5 鉴权

| 端点 | 鉴权 |
|------|------|
| `GET /proxy-pay/{token}` | 公开 |
| `POST /proxy-pay/{token}/pay` | `auth:sanctum` + `password.changed` |
| `POST /orders/{id}/proxy-pay-link` | `auth:sanctum` + `password.changed`，仅订单本人 |

### 2.6 错误码

| code | 异常 | 场景 |
|------|------|------|
| 404 | `ProxyPayTokenNotFoundException` | token 不存在 |
| 42204 | `OrderNotPayableException` | 代付订单走自付 / 订单不可支付 |
| 42205 | `ProxyPayLinkExpiredException` | 链接已过期 |
| 42206 | `OrderNotProxyPayException` | 非代付订单请求生成链接 |

## 4. API 端点

### 公开

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/v1/proxy-pay/{token}` | 代付预览（订单号、金额、下单人、有效期、是否可付） |

### 员工（`auth:sanctum` + `password.changed`）

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/v1/orders/{id}/proxy-pay-link` | 生成代付链接；返回 `url`、`token`、`expires_at` |
| POST | `/api/v1/proxy-pay/{token}/pay` | 代付人发起支付；body: `channel?`, `openid?` |
